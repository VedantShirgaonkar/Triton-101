## Part 2: The Kernel Function

### Lines 8-24: The `@triton.jit` Decorator and Kernel Definition

```python
@triton.jit
def add_kernel(
    x_ptr,
    y_ptr,
    output_ptr,
    n_elements,
    BLOCK_SIZE: tl.constexpr
    ):
```

#### Understanding `@triton.jit`

**What it does**: Decorator that marks this function as a GPU kernel to be JIT-compiled to CUDA.

**Triton Abstraction**:
```python
# What you write (Triton code):
@triton.jit
def add_kernel(x_ptr, y_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    # GPU code here
    pass

# What Triton compiler generates (CUDA equivalent):
__global__ void add_kernel_compiled(float* x, float* y, float* output, 
                                     int n_elements) {
    // Auto-generated CUDA code
    // Handles thread organization, warps, etc.
}
```

**Key Points**:
- Code inside `@triton.jit` runs ON THE GPU (not on CPU)
- Triton compiles it to optimized CUDA
- The function is never called directly—it's compiled and cached

#### Understanding Kernel Parameters

**`x_ptr, y_ptr, output_ptr`** - Pointers to tensor memory

```python
# These are NOT the actual tensors!
# They're pointers to GPU memory addresses

# When you call: add_kernel[grid](x, y, output, ...)
# Triton does: x_ptr = address_of(x), y_ptr = address_of(y), etc.

# Inside kernel:
x = tl.load(x_ptr + offsets)  # Load from that address
```

**Why pointers?**
- Triton needs raw memory addresses (CUDA requirement)
- Allows random access to any part of the tensor
- Different from Python where you work with objects

**CUDA Connection**:
```cuda
// In CUDA you explicitly manage pointers:
__global__ void kernel(float* x_ptr, float* y_ptr, float* output_ptr) {
    // Access memory directly
    float value = x_ptr[idx];
}

// Triton abstracts this away:
@triton.jit
def kernel(x_ptr, y_ptr, output_ptr, ...):
    # Still pointers under the hood, but hidden from you
```

**`n_elements`** - Total number of elements to process

```python
# If x shape is (256,000,), then n_elements = 256,000
# This helps handle boundary cases where grid doesn't divide evenly
```

**`BLOCK_SIZE: tl.constexpr`** - Compile-time constant

#### Understanding `tl.constexpr`

**What it does**: Tells Triton "this is a compile-time constant—embed its value in the kernel"

**Why it matters**:

```python
@triton.jit
def add_kernel(..., BLOCK_SIZE: tl.constexpr):
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    # If BLOCK_SIZE=1024, Triton generates:
    # offsets = block_start + tl.arange(0, 1024)
    
    # If BLOCK_SIZE=2048, Triton generates DIFFERENT CODE:
    # offsets = block_start + tl.arange(0, 2048)
    
    # These are TWO DIFFERENT KERNELS (separately compiled and cached)
```

**Triton Specialization**:
```python
# Call 1: with BLOCK_SIZE=1024
add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=1024)
# Triton compiles: kernel_1024.ptx (NVIDIA intermediate representation)
# Caches it for future use

# Call 2: with BLOCK_SIZE=1024
add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=1024)
# Reuses cached kernel_1024.ptx (no recompilation)

# Call 3: with BLOCK_SIZE=2048
add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=2048)
# Triton compiles: kernel_2048.ptx (different kernel!)
# Caches separately
```

**CUDA Connection**:
```cuda
// In CUDA you'd have to write multiple versions:
template <int BLOCK_SIZE>
__global__ void add_kernel() {
    // ... different code for different BLOCK_SIZE
}

// Instantiate for different sizes:
add_kernel<1024><<<grid, threads>>>(x, y, output);
add_kernel<2048><<<grid, threads>>>(x, y, output);

// Triton does this automatically!
```

**Why use `tl.constexpr`?**
1. Compile-time optimization: Triton knows exact loop bounds
2. Generates optimal code for specific BLOCK_SIZE
3. Fixes array sizes (e.g., `tl.arange(0, BLOCK_SIZE)` must be known at compile time)

**GPU Hardware Layer**:

### Line 32: `mask = offsets < n_elements`

```python
mask = offsets < n_elements
```

**What it does**: Creates a boolean array marking valid indices

**Why it's needed**:
```python
# Example: n_elements = 256,001, BLOCK_SIZE = 1024
# Last program processes: offsets = [256000, 256001, 256002, ..., 257023]
# But indices 256001 and beyond are INVALID!

# mask = offsets < 256001
# Result: [True, False, False, ..., False]
#         (only first element is valid)

# mask = [True, True, ..., True, False, False, ..., False]
#        256 True values (valid indices)
```

**Why we need this**:
- GPU cannot dynamically allocate or skip loads
- Must load fixed amounts, but need to ignore garbage data
- Mask tells hardware: "Load all, but only use these values"

**Triton Abstraction → CUDA**:
```python
# Triton:
mask = offsets < n_elements
x = tl.load(x_ptr + offsets, mask=mask, other=None)

# CUDA equivalent:
bool mask[BLOCK_SIZE];
for (int i = 0; i < BLOCK_SIZE; i++) {
    mask[i] = (offsets[i] < n_elements);
}

float x[BLOCK_SIZE];
for (int i = 0; i < BLOCK_SIZE; i++) {
    if (mask[i]) {
        x[i] = x_ptr[offsets[i]];
    } else {
        x[i] = 0.0f;  // Default value (in our case other=None → undefined)
    }
}
```

### Line 38-39: `tl.load()` - The Key Memory Operation

```python
x = tl.load(x_ptr + offsets, mask=mask, other=None)
y = tl.load(y_ptr + offsets, mask=mask, other=None)
```

#### Understanding `tl.load`

**What it does**: Loads data from GPU memory (HBM) into on-chip memory

**Triton Abstraction → CUDA**:
```python
# Triton (simple):
x = tl.load(x_ptr + offsets, mask=mask, other=None)

# CUDA equivalent:
float x[BLOCK_SIZE];
for (int idx in offsets) {
    if (is_valid) {
        x[idx - block_start] = x_ptr[idx];  // Load from HBM
    } else {
        x[idx - block_start] = NaN;  // other=None means undefined
    }
}
```

**Parameters**:
- `x_ptr + offsets`: Base address + array of offsets
  - If `x_ptr` = 0x1000 (memory address)
  - And `offsets` = [0, 1, 2, ..., 1023]
  - Accesses: [0x1000, 0x1004, 0x1008, ..., 0x1FFC]

- `mask=mask`: Only load where mask is True
  - If mask[i] = False, skip this index
  - Prevents out-of-bounds access

- `other=None`: What to put in x[i] if mask[i]=False
  - `None`: Undefined (don't care)
  - `0.0`: Fill with 0
  - Could be anything—doesn't matter since we won't use it

**GPU Memory Access Pattern**:

```
HBM (Global Memory):
┌─────────────────────────────────────────┐
│ x_ptr→ [x[0] | x[1] | x[2] | ... | x[n]]│
└─────────────────────────────────────────┘
 ↓         loads happen here

GPU Caches:
┌─────────────────────────────────────────┐
│ L2 Cache: Cached by hardware            │
├─────────────────────────────────────────┤
│ L1 Cache: Per-SM automatic cache        │
├─────────────────────────────────────────┤
│ Registers: Thread-local fast storage    │
│ x = [x[0], x[1], x[2], ...] (in regs)   │
└─────────────────────────────────────────┘
```

**SIMD Vectorization**:
```
Triton compiles tl.load to vectorized CUDA:

Instead of:
x[0] = load from address[0]
x[1] = load from address[1]
x[2] = load from address[2]
...
x[31] = load from address[31]
(32 separate loads, slow)

Triton generates:
Load 128 bytes (4 floats × 32 threads) in one coalesced operation
(Much faster!)
```

**Coalesced Memory Access** (GPU-level optimization):
```
All 32 threads in a warp accessing next consecutive addresses:
Thread 0: Load from address 0x1000
Thread 1: Load from address 0x1004
Thread 2: Load from address 0x1008
...
Thread 31: Load from address 0x107C

GPU hardware combines these into ONE 128-byte read from HBM
Speed: ~1 cycle for all 32 loads (instead of 32 cycles sequential)

Without coalescing (bad):
Thread 0: Load from address 0x2000 (random addresses)
Thread 1: Load from address 0x5000
...
Result: 32 separate HBM accesses (~30-50 cycles per access)
```

#### Understanding `triton.cdiv`

**What it does**: Ceiling division (round up division)

**Definition**:
```python
def cdiv(a, b):
    """Ceiling division: a / b rounded up"""
    return (a + b - 1) // b
```

**Example**:
```python
triton.cdiv(256, 64)   # (256 + 64 - 1) // 64 = 319 // 64 = 4   ✓ Correct!
# Normal division: 256 / 64 = 4 (works here)

triton.cdiv(260, 64)   # (260 + 64 - 1) // 64 = 323 // 64 = 5   ✓ Correct!
# Normal division: 260 / 64 = 4.0625 → 4 (WRONG! Would lose 4 elements)

# With ceiling division: ensures ALL elements are covered
```

**Why ceiling division?**

```python
# n_elements = 256,001
# BLOCK_SIZE = 1024

# Normal division: 256,001 / 1024 = 249.9990... → 249 blocks
# Missing: 1 element!

# Ceiling division: (256,001 + 1024 - 1) / 1024 = 250 blocks
# All 256,001 elements covered

# The last block will process [256000:257024]
# Elements beyond 256001 are marked invalid by the mask
```

**The complete lambda**:
```python
grid = lambda meta : (triton.cdiv(n_elements, meta['BLOCK_SIZE']), )

# At kernel launch:
# n_elements = 256,000
# meta['BLOCK_SIZE'] = 1024
# Result: grid = (250,)  ← 250 programs

# Triton will launch:
# For PID in range(250):
#     launch this program on an available SM
```

**What it does**: Launches the kernel on GPU

**Triton Syntax**:
```python
# kernel[grid](args)

# breakdown:
kernel      # The @triton.jit function
[grid]      # How many programs to launch
(args)      # Arguments to pass to kernel
```

**GPU Launch Process**:

```
Step 1: Triton Compiler
├─ Receives: @triton.jit function + kernel_args
├─ Checks cache: Has kernel@BLOCK_SIZE=1024 been compiled?
└─ If not: Compile to CUDA → Optimize → Cache

Step 2: NVIDIA Driver
├─ Receives compiled kernel (PTX bytecode)
├─ Compiles to GPU instructions (SASS)
├─ Loads into GPU instruction cache

Step 3: GPU Scheduler
├─ Receives grid = (250,)
├─ Assigns blocks to SMs:
│  Block 0 → SM 0
│  Block 1 → SM 1
│  ...
│  Block 127 → SM 127
│  Block 128 → SM 0 (after Block 0 finishes)
│  ...
└─ Execution begins

Step 4: Each SM executes its block
├─ Creates threads (Triton chooses thread count automatically)
├─ Organizes into warps (32 threads per warp)
├─ Executes kernel code
└─ Returns control to host when grid completes
```

**Data Flow**:
```
PyTorch tensors (x, y, output)
    ↓
Triton extracts data pointers
    ↓
Kernel receives pointers
    ↓
Inside kernel: tl.load from pointers
    ↓
Process data
    ↓
tl.store back to pointers
    ↓
Kernel completes
    ↓
Host continues (PyTorch returns tensor result)
```

#### `atol` and `rtol` Parameters

**`atol`** = Absolute Tolerance

```python
atol=1e-3  # 0.001

# Allows element-wise absolute difference up to 0.001
# If |X_tri[i] - X_ref[i]| > 0.001 → FAILS

# Used for small numbers:
# X_tri[0] = 0.0001, X_ref[0] = 0.0000
# diff = 0.0001, atol = 0.001
# 0.0001 < 0.001 → PASS ✓
```

**`rtol`** = Relative Tolerance

```python
rtol=1e-3  # 0.001 (0.1%)

# Allows relative (percentage) difference up to 0.1%
# If |X_tri[i] - X_ref[i]| / |X_ref[i]| > 0.001 → FAILS

# Used for large numbers:
# X_tri[0] = 100.1, X_ref[0] = 100.0
# diff = 0.1, X_ref = 100.0
# relative_diff = 0.1 / 100.0 = 0.001 (0.1%)
# 0.001 < 0.001 → PASS (borderline)
```

**Combined Check**:
```python
# PASSES if:
# abs_diff < atol OR rel_diff < rtol

# Example:
atol = 0.001, rtol = 0.001

Case 1: X_tri[0] = 1.0001, X_ref[0] = 1.0
abs_diff = 0.0001, rel_diff = 0.0001
atol check: 0.0001 < 0.001 ✓ → PASS

Case 2: X_tri[0] = 5.01, X_ref[0] = 5.0
abs_diff = 0.01, rel_diff = 0.002
atol check: 0.01 > 0.001 ✗
rtol check: 0.002 > 0.001 ✗ → FAIL

Case 3: X_tri[0] = 1000.1, X_ref[0] = 1000.0
abs_diff = 0.1, rel_diff = 0.0001
atol check: 0.1 > 0.001 ✗
rtol check: 0.0001 < 0.001 ✓ → PASS
```

**Why both tolerances?**
- `atol`: Handles small numbers (where absolute error matters)
- `rtol`: Handles large numbers (where percentage error matters)

**GPU floating-point errors**:
```python
# CPU computation:
a + b = 1.00000000

# GPU computation (slightly different rounding):
a + b = 0.99999999

# Difference: 1e-8 (too strict with atol=1e-10)
# But relative: 1e-8 / 1.0 = 1e-8 (catches with rtol=1e-7)
```

```python
def benchmark(size, provider):
    x = torch.randn(size, device=DEVICE, dtype=torch.float32)
    y = torch.randn(size, device=DEVICE, dtype=torch.float32)
    
    quantiles = [0.5, 0.05, 0.95]
    
    if provider == 'torch':
        ms, min_ms, max_ms = triton.testing.do_bench(
            lambda: x + y, 
            quantiles=quantiles
        )
    if provider == 'triton':
        ms, min_ms, max_ms = triton.testing.do_bench(
            lambda: add(x, y), 
            quantiles=quantiles
        )
    
    gbps = lambda ms: 3 * x.numel() * x.element_size() * 1e-9 / (ms * 1e-3)
    
    return gbps(ms), gbps(max_ms), gbps(min_ms)
```

#### Understanding `triton.testing.do_bench()`

```python

```

**What it does**: Benchmarks a function and returns quantile timing

**How it works**:
```python
# Simplified:
def do_bench(fn, quantiles=[0.5, 0.05, 0.95]):
    times = []
    
    # Warm up GPU
    for _ in range(100):
        fn()
    
    # Measure actual performance
    for _ in range(1000):
        t_start = time.time()
        fn()
        t_elapsed = time.time() - t_start
        times.append(t_elapsed)
    
    # Calculate quantiles
    sorted_times = sorted(times)
    ms = sorted_times[n * 0.5]      # Median (50th percentile)
    min_ms = sorted_times[n * 0.05] # 5th percentile (best case)
    max_ms = sorted_times[n * 0.95] # 95th percentile (worst case)
    
    return ms, min_ms, max_ms  # In milliseconds
```

**Why quantiles?**
```python
# Instead of min/max (which are outliers):
times = [4.2ms, 4.1ms, 4.3ms, 100ms, 4.2ms, ...]
min = 4.1ms ✓ Good
max = 100ms ✗ Outlier! (GC happened, or other interference)

# Quantiles give representative performance:
median (50th percentile) = 4.2ms (typical)
5th percentile = 4.1ms (best case, no interference)
95th percentile = 4.3ms (allows some jitter, ignores outliers)
```

#### Understanding the GBps Calculation

```python
gbps = lambda ms: 3 * x.numel() * x.element_size() * 1e-9 / (ms * 1e-3)
```

Break it down:

**Numerator: `3 * x.numel() * x.element_size() * 1e-9`**
```python
# Vector addition reads x, reads y, writes output = 3 accesses
# x.numel() = total elements (e.g., 256,000)
# x.element_size() = bytes per element (4 for float32)
# 1e-9 = convert bytes to GB

# Example:
# 3 * 256000 * 4 * 1e-9 = 0.003 GB (3 MB of data moved)
```

**Denominator: `ms * 1e-3`**
```python
# ms = time in milliseconds
# 1e-3 = convert to seconds

# Example:
# ms = 4 milliseconds
# 4 * 1e-3 = 0.004 seconds
```

**Complete Calculation**:
```python
# Example:
size = 256,000 floats
time = 4 milliseconds

data_volume = 3 * 256000 * 4 bytes = 3,072,000 bytes = 3 MB
time = 0.004 seconds = 4 ms

bandwidth = 3 MB / 0.004 s = 750 MB/s = 0.75 GB/s

# In code:
gbps(4) = 3 * 256000 * 4 * 1e-9 / (4 * 1e-3)
        = 0.003072 / 0.004
        = 0.768 GB/s
```

**Why measure GB/s for vector add?**
- Vector addition is **memory-bound** (limited by memory bandwidth)
- Computation is trivial (1 add per 3 memory operations)
- Metric shows how efficiently we move data
- GPU peak bandwidth: ~500-1000 GB/s depending on model
- Our implementation typically achieves 80-90% of peak


Returns 3 values:
- Median throughput (typical case)
- Max throughput (best case)
- Min throughput (worst case)

## Summary: From Triton to CUDA to GPU Hardware

```
Your Python Code (Triton)
├─ @triton.jit: Marks GPU function
├─ tl.program_id(): Which block am I?
├─ tl.arange(): Create offset arrays
├─ tl.load(): Load from memory
├─ tl.store(): Write to memory
└─ tl.constexpr: Compile-time constants

        ↓ Triton Compiler

Generated CUDA Code
├─ __global__ kernels
├─ Thread organization
├─ Warp-level instructions
├─ Memory coalescing
└─ Warp synchronization

        ↓ NVIDIA PTX Compiler

GPU Binary (SASS instructions)
├─ Load instructions (cache-aware)
├─ FMA (multiply-add) instructions
├─ Store instructions
└─ Warp scheduling directives

        ↓ GPU Execution

Hardware Execution
├─ SMs load kernel
├─ Blocks assigned to SMs
├─ Threads organized into warps
├─ Warps execute in parallel
├─ Memory accesses coalesced
└─ Results written back to HBM

        ↓ Synchronization

Result returned to Host (CPU)
└─ PyTorch tensor with results
```
