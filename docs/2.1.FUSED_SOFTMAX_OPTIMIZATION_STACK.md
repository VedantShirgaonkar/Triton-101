# Fused Softmax Optimization Stack

A complete explanation of why fused softmax kernels achieve 8-10x speedup over naive implementations.

---

## 1. Memory Overhead Reduction: Naive vs Fused

### The Problem: Naive Softmax (5 Kernels)

```
Naive Softmax Pipeline:
┌──────────────────────────────────────────┐
│ Kernel 1: x_max = max(x)                 │
│ Read x (16 floats) → Write x_max (4)     │
├──────────────────────────────────────────┤
│ Kernel 2: z = x - x_max                  │
│ Read x (16) + Read x_max (4) → Write z   │
├──────────────────────────────────────────┤
│ Kernel 3: numerator = exp(z)             │
│ Read z (16) → Write numerator (16)       │
├──────────────────────────────────────────┤
│ Kernel 4: denominator = sum(numerator)   │
│ Read numerator (16) → Write denom (4)    │
├──────────────────────────────────────────┤
│ Kernel 5: output = numerator/denominator │
│ Read numerator (16) + Read denom (4)     │
│ → Write output (16)                      │
└──────────────────────────────────────────┘

Total HBM traffic: 224 MB read + 112 MB write = 336 MB
Data to process: 32 MB
Amplification: 336 MB / 32 MB = 10.5x !!!

Breakdown per operation:
├─ max:    32 MB (read x) + 16 KB (write x_max)
├─ sub:    96 MB (read x + x_max) + 32 MB (write z)
├─ exp:    32 MB (read z) + 32 MB (write numerator)
├─ sum:    32 MB (read numerator) + 16 KB (write denom)
└─ divide: 96 MB (read numerator + denom) + 32 MB (write output)
```

**Each intermediate result bounces to HBM**, then gets read again!

### The Solution: Fused Softmax (1 Kernel)

```
Fused Softmax Pipeline:
┌──────────────────────────────────────────────────────────┐
│ Single Kernel: All operations in GPU registers           │
│                                                          │
│ for each row:                                            │
│   ├─ Load row from HBM (16 MB)                          │
│   ├─ max = max(row)          [stays in registers]       │
│   ├─ sub = row - max         [stays in registers]       │
│   ├─ exp = exp(sub)          [stays in registers]       │
│   ├─ sum = sum(exp)          [stays in registers]       │
│   ├─ output = exp / sum      [stays in registers]       │
│   └─ Write output to HBM (16 MB)                        │
└──────────────────────────────────────────────────────────┘

Total HBM traffic: 32 MB read + 32 MB write = 64 MB
Data to process: 32 MB
Amplification: 64 MB / 32 MB = 2x (much better!)

Key insight: All intermediate results stay in FAST registers
            No intermediate HBM traffic
            Only final result written once
```

**Performance Impact**:
- Naive: 336 MB HBM traffic @ 900 GB/s = 0.37 ms (+ 1 ms overhead) = 1.4 ms total
- Fused: 64 MB HBM traffic @ 900 GB/s = 0.07 ms (+ 0.1 ms overhead) = 0.17 ms total
- **Speedup: 8.2x**

## 2. Coalesced Memory Access

### Why It Matters

32 threads can load memory **32x faster** if they access consecutive addresses instead of random ones.

### Non-Coalesced (SLOW)

```
32 threads, random addresses:

Thread 0:  Load from 0x5000 ─┐
Thread 1:  Load from 0x9000  ├─ Each thread makes separate HBM request
Thread 2:  Load from 0x2000  │  30 separate latencies!
...                          │  TOTAL: 32 × 300 = 9600 cycles
Thread 31: Load from 0x8000 ─┘
```

### Coalesced (FAST)

```
32 threads, consecutive addresses:

Thread 0:  Load from 0x1000 ─┐
Thread 1:  Load from 0x1004  │ Consecutive!
Thread 2:  Load from 0x1008  │ GPU detects pattern
...                          │ Fetches 1 cache line (128 bytes)
Thread 31: Load from 0x107C ─┘ TOTAL: 300 cycles (38x faster!)

GPU hardware combines: "These are consecutive memory addresses"
Fetches entire range in ONE HBM access
All 32 threads get data in parallel
```

### Making it Happen in Triton

```python
@triton.jit
def kernel(x_ptr, y_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    PID = tl.program_id(axis=0)
    block_start = PID * BLOCK_SIZE
    
    # Key: tl.arange creates CONSECUTIVE offsets
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    # offsets = [base, base+1, base+2, ..., base+BLOCK_SIZE-1]
    
    # Result: COALESCED access ✓
    x = tl.load(x_ptr + offsets, ...)
```

**Triton automatically generates coalesced loads** when you use `tl.arange()`

## 3. Persistent Kernel Loop

### Problem: Load Imbalance (Non-Persistent)

```
Configuration:
├─ GPU: 128 SMs
├─ Launch: 4096 blocks (one per row)
└─ Each block: processes ONE row

Timeline:
T=0ms:    All 128 SMs loaded (blocks 0-127 run)
          Blocks 128-4095 waiting in queue...

T=1ms:    Some blocks finish (not all!)
          SM 0 idle, SM 2 idle, but SM 50 still busy
          Load imbalance!

Result: Poor GPU utilization (~60%)
        Many SMs sit idle waiting for others
```

### Solution: Persistent Kernel Loop

```
Configuration:
├─ GPU: 128 SMs
├─ Launch: 256 blocks (2 per SM average)
└─ Each block: processes MULTIPLE rows

Block 0: for row_idx in [0, 256, 512, ..., 3840]
Block 1: for row_idx in [1, 257, 513, ..., 3841]
Block 2: for row_idx in [2, 258, 514, ..., 3842]
...

Timeline:
T=0ms:    All 256 blocks start immediately
          (No queue! All blocks active!)

T=1ms:    Block 0 finishes first row
          Immediately grabs next row (256)
          Block 1 still computing row 1
          No waiting!

Result: Perfect load balancing!
        GPU utilization: 99%
        Every SM always has work
```

### Code Implementation

```python
# Non-persistent (bad):
@triton.jit
def kernel(input_ptr, output_ptr, row_id, ...):
    row = tl.load(input_ptr + row_id * stride, ...)
    output = softmax(row)
    tl.store(output_ptr + row_id * stride, output)

grid = (4096, )  # 4096 blocks for 4096 rows


# Persistent (good):
@triton.jit
def kernel(input_ptr, output_ptr, ...):
    PID = tl.program_id(0)
    row_step = tl.num_programs(0)  # e.g., 256 total blocks
    
    # Loop: each block handles multiple rows
    for row_idx in tl.range(PID, n_rows, row_step, num_stages=4):
        row = tl.load(input_ptr + row_idx * stride, ...)
        output = softmax(row)
        tl.store(output_ptr + row_idx * stride, output)

grid = (256, )  # 256 blocks, each processes ~16 rows
```

**Key**: Blocks work at their own pace, grabbing new work when ready. No idle time!

## 4. Multi-Stage Pipelining (num_stages)

### Problem: Memory Latency Stalls

```
Single-stage (no pipelining):

Cycle 0-5:    Compute row 0
Cycle 6-305:  Load row 1 (STALL! 300 cycles waiting)
              GPU cores are IDLE
Cycle 306-311: Compute row 1
Cycle 312-611: Load row 2 (STALL! Again!)
...

For 100 rows:
Total time = 100 loads × 300 cycles + 100 computes × 5 cycles
           = 30,000 + 500 = 30,500 cycles
GPU efficiency: 500 / 30,500 = 1.6% (TERRIBLE!)
```

### Solution: 4-Stage Pipeline

```
While waiting for load, do OTHER work!

Time  Load      Compute   Store
 0:   Load i0   —         —
 1:   Load i1   Compute0  —
 2:   Load i2   Compute1  Compute0
 3:   Load i3   Compute2  Compute1  Store0
 4:   Load i4   Compute3  Compute2  Store1
 5:   Load i5   Compute4  Compute3  Store2
 6:   Load i6   Compute5  Compute4  Store3
...
300:  —         Compute299 ...      ...
      (i0 data arrives after 300 cycles)
      But we've been computing other iterations!

Result: Memory latency is HIDDEN
        Cores always have compute work
        GPU efficiency: 80-90% (vs 1.6%)
```

### In Triton Code

```python
for row_idx in tl.range(PID, n_rows, row_step, num_stages=4):
    #                                          ↑ This parameter!
    
    row = tl.load(input_ptr + row_idx * stride, ...)
    output = softmax(row)
    tl.store(output_ptr + row_idx * stride, output)
```

**num_stages parameter**:
- 1: No pipelining (sequential)
- 2: 2 iterations can overlap
- 4: 4 iterations can overlap (typical)
- 8: 8 iterations can overlap (aggressive)

**Triton compiler automatically**:
1. Generates prologue code (fill pipeline with first 4 iterations)
2. Main loop (maintains 4 active iterations)
3. Epilogue code (drain remaining iterations)

All transparent to you!

## 5. Warmup: Compilation vs Execution

### Without Warmup (Naive)

```python
kernel = _softmax_kernel
output = kernel[grid](x, y, ...)

Timeline:
T0:       kernel[grid](...) called
T0-500ms: COMPILATION (first time seeing this config)
          ├─ Parse Triton code
          ├─ Lower to LLVM IR
          ├─ Optimize
          ├─ Convert to PTX
          └─ Cache result

T500ms:   EXECUTION (now run it)
          ├─ Load kernel onto GPU
          ├─ Launch grid
          └─ Process data (~0.5 ms)

T500.5ms: Done! Return to Python

Total: ~500 ms (mostly compilation!)
```

### With Warmup (Better)

```python
kernel = _softmax_kernel.warmup(
    x, y, x.stride(0), y.stride(0),
    n_rows, n_cols,
    BLOCK_SIZE=1024, num_stages=4, num_warps=8,
    grid=(1,)  # Dummy grid for compilation
)
kernel.init_handles()

# Query metadata BEFORE execution
n_regs = kernel.n_regs                    # e.g., 40
sram = kernel.metadata.shared             # e.g., 8192

# Calculate optimal occupancy
reg_occupancy = 65536 // (40 * 32 * 8)    # = 6
sram_occupancy = 98304 // 8192            # = 12
programs_per_sm = min(6, 12)              # = 6
num_programs = min(128 * 6, n_rows)       # = 768

# Launch with optimal grid
output = kernel[(768,)](x, y, ...)

Total time: 500 ms compilation + 0.5 ms execution
            (but with PERFECT occupancy!)
```

### Why Warmup Matters

```
Warmup separates two things:

1. COMPILATION (T0-500ms):
   ├─ Parse + optimize Triton code
   ├─ Generate PTX (NVIDIA IR)
   └─ Cache for reuse

2. METADATA QUERY (T500ms):
   ├─ kernel.n_regs = ? (compiler tells us)
   ├─ kernel.metadata.shared = ? (compiler tells us)
   └─ Calculate optimal grid BEFORE launching

Without warmup:
├─ Compilation happens at first kernel[grid]() call
├─ Can't query metadata until after compilation
├─ Have to guess at grid size
└─ Might waste GPU resources

With warmup:
├─ Compilation front-loaded (done once)
├─ Can query metadata immediately
├─ Calculate PERFECT occupancy
└─ Launch with optimal grid
```

**Warmup benefits**:
- Know exact register usage per thread
- Know exact SRAM usage per block
- Calculate exactly how many blocks fit per SM
- Achieve 85-95% GPU utilization vs 15% without

## Key Concepts at a Glance

| Concept | What | Why |
|---------|------|-----|
| **Coalesced Access** | All threads load consecutive memory | 32x faster memory bandwidth |
| **Persistent Loop** | Block processes multiple rows | Load balancing, no idle SMs |
| **Pipelining** | Overlap Load/Compute/Store stages | Hide 300-cycle memory latency |
| **In-Register Compute** | Keep intermediates in registers | Avoid 10 HBM passes |
| **Warmup** | Query metadata before execution | Optimal occupancy calculation |
| **Fused Kernel** | Single kernel, all ops together | 9x less memory traffic |
